Abstract

The DRDO GPF Management System is a secure, full-stack web application developed to digitise and streamline General Provident Fund (GPF) operations for employees of the Defence Research and Development Organisation. The system replaces manual, paper-based GPF processes with an integrated digital platform covering employee registration, JWT-secured authentication, GPF account management, subscription tracking, advance applications (temporary and final withdrawals), sanction workflows, and report generation.

Built on a modern two-tier architecture — a React 19 single-page application communicating with a Spring Boot 4 RESTful backend, backed by an Oracle XE relational database — the system enforces security via JSON Web Tokens and manages all schema changes through versioned Flyway migrations. It supports 9 UI themes, role-aware navigation guards, and a fully responsive layout.

2

Introduction

The General Provident Fund is a mandatory savings scheme for central government employees in India. Managing GPF records manually is error-prone, time-consuming, and difficult to audit. This system addresses those pain points:

🔐

Secure Authentication

JWT login with password expiry, security questions, and self-service password recovery.

📋

GPF Account Management

Create and manage GPF accounts, track account numbers, designations, and service dates.

💰

Subscription Tracking

Record monthly GPF subscriptions, DV numbers, and year-wise closing balances.

📤

Advance Applications

Apply for temporary advances or final withdrawals with purpose selection and amount tracking.

✅

Sanction Workflow

Admin sanctions applications, sets instalment amounts, and tracks repayment.

📊

Reports

Generate per-employee GPF reports covering subscriptions, advances, and balances.

The system follows a classic three-layer client-server architecture with clear separation between presentation, business logic, and data layers.

Communication Details

  • React frontend runs on http://localhost:5173 (Vite dev server)
  • All API calls target http://localhost:8081/api/** with Bearer JWT in Authorization header
  • Spring Security validates JWT on every protected endpoint via a stateless filter chain
  • Flyway executes versioned SQL migration scripts on startup to keep Oracle schema in sync
  • CORS configured to allow all localhost:* origins during development

Route Guards

Security

🔑

JWT Authentication

HMAC-SHA256 signed tokens. 1-hour expiry. Sent as Bearer token in Authorization header on every protected request.

🔒

BCrypt Passwords

All passwords hashed with BCrypt before storage. Never stored or transmitted in plain text.

⏰

Password Expiry

Configurable expiry in days. User is forced to change password on next login after expiry.

🛡️

Route Guards

PrivateRoute blocks unauthenticated access. PublicRoute prevents logged-in users from navigating back to login.

❓

Security Questions

Two security questions set at registration. Used for self-service password recovery without email dependency.

🌐

CORS Policy

Configured to allow localhost:* origins during development. Must be restricted to specific origins in production.

Password Rules

  • Minimum 8 characters, maximum 12 characters
  • At least 1 uppercase letter
  • At least 1 numeric digit
  • At least 1 special character (!@#$%^&* etc.)

User Flows

Login Flow

1

Open Application

Navigate to http://localhost:5173. If already logged in, PublicRoute redirects to /dashboard automatically.

2

Enter Credentials

Enter User ID and Password on the login form.

3

JWT Issued

Backend validates credentials, returns JWT token + user session data (userId, username, workStatus, dob, numericId).

4

Session Stored

Token and user data stored in both localStorage and sessionStorage for persistence.

5

Redirect to Dashboard

User lands on /dashboard. Browser back button no longer returns to login page.

GPF Application Flow

1

Open User Application GPF

From dashboard, click "User Application GPF".

2

Auto-fill Data

Phone number and basic pay fetched from GPF table. Closing balance auto-filled from GPF Years (latest year).

3

Select Loan Type & Purpose

Choose Temporary or Final. Purpose dropdown populates from GPF_PURPOSE_E or GPF_PURPOSE_F accordingly.

4

Fill & Submit

Enter amount, enclosures, previous advance details. Click Save — POST to /api/gpf-usr-details/create.

5

Record Created

Application saved to GPF_USR_DETAILS. Appears in "Not Yet Approved" table.

6

Admin Sanctions

Admin reviews in Temporary Advance / Final Withdrawal screens and creates a sanction record in GPF_SANCTION_DETAILS.

Setup & Run

Prerequisites

  • Java 17+
  • Node.js 18+ and npm
  • Oracle Database XE running on localhost:1521 with SID xe
  • Oracle user aditya with password aditya2468 (or update application.properties)

Start Backend

cd drdo-BackEnd/loginregister

# Windows
mvnw.cmd spring-boot:run

# Linux / Mac
./mvnw spring-boot:run

# Server starts on http://localhost:8081
# Flyway migrations run automatically on startup

Start Frontend

cd drdo-FrontEnd

npm install        # first time only
npm run dev        # starts on http://localhost:5173

npm run build      # production build → dist/

Environment Config

Frontend reads API base URL from drdo-FrontEnd/.env:

VITE_API_BASE_URL=http://localhost:8081

Defaults to http://localhost:8081 if not set.
